## Details per phase

### Phase 5a — `pssz_view<T, F>`

New header `libraries/psio1/cpp/include/psio/pssz_view.hpp`. Mirrors
`ssz_view` but:
- Format-parameterized (internal 8-bit / pssz16 / pssz32 offset widths)
- Reflected containers skip the `F::header_bytes` extensibility header
  automatically
- `Option<T>` accessor honors the min-size-gated selector rule (no
  selector when inner has min > 0)

Tests in `pssz_tests.cpp`:
- `pssz: view primitive and string`
- `pssz: view reflected DWNC struct (BPoint)`
- `pssz: view reflected extensible struct (UserProfile)` — reads every
  field of a 7-field optional-carrying struct via `field<I>()`

All field-access patterns are span-arithmetic; no allocations.

### Phase 5b — `bounded<T, N>`

Added `psio1::bounded<T, N>` in `bounded.hpp` — a generic wrapper that
takes any container-like `T` (String, Vec, Array, …) with a compile-time
cap. Provides `storage()`, `size()`, `empty()`, `data()`, and a
SFINAE-gated `view()` for string-T.

**Legacy `bounded_string<N>` / `bounded_list<T,N>` / `bounded_bytes<N>`
classes are retained** so existing format specializations and all 703
tests keep passing. The intended future path is a separate, staged
migration that rewrites each format's specializations to operate on
`bounded<T, N>` generically, then replaces the legacy classes with
aliases. Too disruptive to do overnight without extensive risk.

Test: `generic bounded<T,N> basic usage` — covers construction, bound
enforcement via `REQUIRE_THROWS` on overflow, storage access, and the
`is_bounded_v<T>` detection trait.

### Phase 5c — Rust pssz port

New module `libraries/psio1/rust/psio/src/pssz.rs` (~350 lines). API:

```rust
use psio1::pssz::{Pssz32, to_pssz, from_pssz};
let bytes = to_pssz::<Pssz32, _>(&42u32);
let back: u32 = from_pssz::<Pssz32, u32>(&bytes)?;
```

Covers:
- Format traits `Pssz8` / `Pssz16` / `Pssz32` (offset/header widths)
- `PsszPack<F>` / `PsszUnpack<F>` traits with `MIN_ENCODED_SIZE`,
  `IS_FIXED_SIZE`, `FIXED_SIZE` associated constants
- Primitives: u8/u16/u32/u64, i8/i16/i32/i64, f32/f64, bool
- `String`, `Vec<T>`, `Option<T>` with min-size-gated selector rule
- Single-pass backpatching encoder (mirroring the C++ fix)

Tests: 13 total.
- 7 native Rust round-trips (primitives, string, vec fixed/variable,
  option fixed/variable inner, pssz16 narrower offsets)
- **6 cross-validation tests** using hex fixtures generated by the C++
  encoder — Rust decodes and re-encodes to byte-identical output:

```
cpp_pssz32_u32_cross          → 0xDEADBEEF
cpp_pssz32_string_cross        → "hello"
cpp_pssz32_vec_u32_cross       → [1, 2, 3]
cpp_pssz32_vec_string_cross    → ["a", "bc", "def"]
cpp_pssz32_option_u32_cross    → Some(42) / None
cpp_pssz32_option_string_cross → Some("x") / None
```

Not in this bootstrap port (documented as followup in
`pssz-format-design.md`):
- Derive macro for reflected-struct types (would parallel `#[derive(Pack)]`)
- uint128 / uint256 support (blocked on the same C++ → Rust parity gap
  noted in the ext_int issue file)
- Schema IR integration
- Zero-copy view trait

## Known regressions / things to look at next session

- `SensorReading` pssz pack: 51 ns vs frac16 20 ns — the internal 8-bit path has
  small-offset overhead that hurts structs with just 2 variable fields
  in a mostly-fixed body. Worth checking the inlined assembly; may be
  a codegen issue rather than algorithmic.
- `ssz` SensorReading pack regressed from 30 → 47 ns after the
  backpatching fix (interleaved patch+payload writes seem to inhibit
  some vectorization the old 3-pass had). Should revisit: try emitting
  the offset table first, then payloads with explicit patch-on-return.
- `bounded<T, N>` aliasing / migration: decided not to do it overnight.
  Plan: make the legacy classes thin aliases in a separate PR with
  explicit tests for each format.
