## Description

`psio::own<T>` was upgraded to proper RAII (destructor, move-only, `release()`)
in commit `f7b4537`, but the destructor calls a default no-op
`wit_resource_drop<T>` template for every resource type. Until each resource
type specializes this template, `own<T>` still leaks on scope exit — same
behavior as before.

This issue tracks the work to wire up real drop calls for every WIT resource
across both host and guest contexts, and to audit call sites that now need
explicit `std::move` to transfer ownership.

## Why This Matters

In a long-running psiserve process, a single leaked handle per accepted
connection means the handle table fills up and either slows down (if it grows)
or refuses new connections (if it's fixed-size). Every example in the WASI
layer (tcp_echo.cpp and beyond) currently leaks handles on every iteration.

This blocks #0012 (WIT host/guest DB API) because database cursors,
transactions, and table handles cannot leak without breaking long-lived
blockchain nodes and HTTP services.

## Scope

### 1. Specialize `wit_resource_drop<T>` for every WIT resource

**wasi:io**
- `input_stream`, `output_stream`, `pollable`, `io_error`

**wasi:sockets**
- `network`, `tcp_socket`, `udp_socket`
- `incoming_datagram_stream`, `outgoing_datagram_stream`
- `resolve_address_stream`

**wasi:filesystem**
- `descriptor`, `directory-entry-stream`

**wasi:clocks / wasi:cli** — check for any resources

**psiserve / psizam** — `module_handle`, `instance`, etc. (once resource-fied)

### 2. Dual-context specialization

Each resource needs two specializations selected by context (build flag,
namespace, or translation unit):

- **Guest side** (inside the WASM module): specialization calls the
  `[resource-drop]<T>` WASM import for the resource's interface
- **Host side** (in psiserve): specialization calls
  `handle_table<T>.destroy(handle)` on the owning host object

The host case is tricky — `wit_resource_drop<tcp_socket>` needs access to the
`WasiSocketsHost::tcp_sockets` table, which is instance-state, not global.
Options:
- thread-local pointer to the current host (hacky but works)
- pass a host reference through own<T> (changes the ABI shape — breaks
  canonical-ABI u32-only wire format)
- generate a registry keyed by handle range or generation (cleanest but
  more plumbing)

Needs design discussion before implementation.

### 3. Code generation via PSIO_REFLECT

Long-term, specializations should be generated by the PSIO reflection macro,
not hand-written. Every `resource` block in a WIT interface should emit both:
- the C++ resource type (already done)
- the `wit_resource_drop<T>` specialization appropriate to the build context

### 4. Call-site audit for `std::move`

`own<T>` is now move-only. Every existing call site that passes `own<T>`
by value into a function that takes ownership must use `std::move`. Likely
locations:
- `libraries/wasi/cpp/examples/tcp_echo.cpp`
- `libraries/wasi/cpp/include/wasi/0.2.3/*_host.hpp` — any function signature
  taking `own<T>` by value as a "consuming" parameter
- Any test code using WIT resources

### 5. Tests

- A mock resource type with a refcounted/tracking `wit_resource_drop`
  specialization, exercised through scope exit, move, self-move,
  variant-contained, tuple-returned, and exception-path scenarios
- An integration test for tcp_echo that accepts N connections in a loop
  and verifies the host's socket table size stays bounded
- A negative test that verifies the canonical ABI `unwrap` path does not
  double-drop
